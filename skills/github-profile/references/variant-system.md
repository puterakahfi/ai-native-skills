# GitHub Profile Variant System

A variant is an expression contract for the same verified profile strategy. It is not a biography generator, a content template, or permission to replace established brand grammar.

## Layer Model

```text
verified identity + audience + goal + evidence
  → content archetype
  → design foundation
  → brand grammar and preservation locks
  → profile variant
  → GitHub-native composition
  → rendered review
```

The content archetype decides which information roles are needed. The variant decides hierarchy, density, rhythm, alignment, containment, and expression. Foundation and factual integrity apply to every variant.

## Invariants Across Variants

Every variant must preserve:

```text
positioning before technology inventory
proof before decoration
verified project anatomy
one coherent narrative
no fabricated biography, metrics, clients, or outcomes
critical information in semantic text
GitHub rendering and narrow-width continuity
maintainable source
```

A variant fails when it changes facts, hides weak proof with decoration, or uses a different information strategy without an explicit redesign decision.

## Variant Axes

Record the following axes instead of treating a variant name as a magic style word:

```yaml
variant_axes:
  density: sparse | balanced | compact
  alignment: centered-opening | left | mixed-editorial
  expression: restrained | contemporary | expressive
  proof_emphasis: low | medium | high
  thesis_emphasis: low | medium | high
  containment: none | subtle-structural | bounded-modules
  separator_behavior: none | subtle | structural
  visual_assets: none | restrained | showcase
  dynamic_widgets: none | minimal | selective
```

The same variant may resolve differently under different brand locks. For example, `modern-professional` may use subtle separators for one brand and whitespace-only grouping for a line-free brand.

## Phase-One Variants

### `zen-minimalist`

```yaml
density: sparse
alignment: centered-opening
expression: restrained
proof_emphasis: medium
thesis_emphasis: high
containment: none
separator_behavior: none
visual_assets: none
```

Recognition comes from authored thesis, precise hierarchy, purposeful Ma, and quiet editorial cadence.

### `modern-professional`

```yaml
density: balanced
alignment: left
expression: contemporary
proof_emphasis: high
thesis_emphasis: medium
containment: subtle-structural
separator_behavior: subtle
visual_assets: restrained
```

Recognition comes from crisp information hierarchy, compact proof modules, clear actions, and contemporary but durable composition.

### `creative-editorial`

```yaml
density: balanced
alignment: mixed-editorial
expression: expressive
proof_emphasis: medium
thesis_emphasis: high
containment: none-or-selective
separator_behavior: brand-dependent
visual_assets: restrained
```

Recognition comes from authored voice, intentional section naming, editorial pacing, and one coherent expressive device—not random decoration.

## Variant Selection Contract

Record:

```yaml
profile_variant_decision:
  selected: zen-minimalist | modern-professional | creative-editorial | custom
  requested_by_user: true | false
  content_archetype: technical-authority | editorial-personal-brand | product-builder | open-source-maintainer | custom
  audience_fit: <reason>
  goal_fit: <reason>
  proof_fit: <reason>
  brand_fit: <reason>
  resolved_axes: {}
  preservation_locks: []
  prohibited_treatments: []
  rejected_variants: []
```

Do not select a variant before content inventory and brand inspection.

## Multi-Variant Mode

When the user requests several variants:

```text
1. Lock one verified fact set.
2. Lock one content inventory and project proof set.
3. Keep the primary profile goal and desired action stable.
4. Produce materially different expression contracts.
5. Do not create artificial differences by changing claims or adding fake assets.
6. Compare rendered results using the same viewport and content completeness.
```

A valid comparison changes hierarchy, rhythm, density, alignment, proof treatment, and voice while preserving factual truth.

## Brand Override Rules

```text
explicit brand prohibition > variant default
verified brand asset rule > variant asset preference
brand separator rule > variant containment preference
foundation legibility/grouping > brand or variant styling
```

Examples:

```text
line-free brand + modern-professional
  → keep contemporary hierarchy and compact proof
  → remove default separators and card containment

calm brand + creative-editorial
  → use authored labels and mixed cadence
  → reject neon, random emoji, ASCII noise, and visual chaos
```

## Variant Failure Signals

```text
same README with only heading labels renamed
facts or project emphasis changed without strategy approval
modern interpreted as cards, badges, gradients, or dashboards by default
creative interpreted as random symbols, animations, or mixed styles
minimalist interpreted as tiny text and maximum empty space
variant conflicts with brand grammar
all variants share the same hierarchy and rhythm
```

## Render Comparison

For a variant set, verify:

```text
same facts and links
same evidence strength
clear visual distinction between variants
foundation passes in every variant
brand compatibility stated for every variant
narrow-width continuity
light/dark behavior when assets or theme-sensitive treatments exist
```

Do not rank a variant as “best” without the target goal, audience, brand, and rendered evidence.
