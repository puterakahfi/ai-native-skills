# Architecture Review

## The Core Rule

```
No agent, no PR, no generated code passes review
without being checked against the Engineering Contract.
Compiled ≠ Approved.
```

## When to Use

- Before merging any PR (new feature or bugfix)
- After an AI agent generates code
- When a new dependency is proposed
- When folder structure changes
- When a new integration is added
- When "just this once" exceptions are requested

## Prerequisites

1. Load the product's `engineering-contract.yaml` before starting
2. Have the code diff, PR, or implementation artifact ready

---

## Review Checklist

### 1. Stack Compliance

- [ ] Backend framework matches contract?
- [ ] Frontend framework matches contract?
- [ ] Database/ORM matches contract?
- [ ] AI provider follows contract rules?
- [ ] No unauthorized framework introduced?

**Violation signal:** New import/dependency not in contract stack.

---

### 2. Architecture Style Compliance

- [ ] Follows declared architecture style (Clean Architecture, Hexagonal, etc)?
- [ ] Domain modeling approach followed (DDD, etc)?
- [ ] No business logic leaking into the wrong layer?
- [ ] No controller calling repository directly?
- [ ] Domain layer has zero framework/infrastructure imports?
- [ ] Ports (interfaces) defined in domain layer, adapters in infrastructure?
- [ ] Aggregates have single root — external access via root ID only?
- [ ] Value objects are immutable?
- [ ] Domain events named in past tense?
- [ ] Repositories return aggregates, not raw entities or DTOs?
- [ ] No domain logic in application service or controller?
- [ ] Pattern selection justified by forces (not preference)?
- [ ] No domain entity importing infrastructure?

**Violation signal:** Cross-layer imports, fat controllers, anemic domain models, god classes.

---

### 3. Dependency Audit

For every new dependency introduced:

- [ ] Is it in the approved dependency list?
- [ ] Does it duplicate an existing approved library?
- [ ] Was it added because it's "convenient" without contract review?
- [ ] Does it require an ADR?

**Flag immediately:** Any library added without prior contract approval.

---

### 4. Folder Structure

- [ ] New files placed in correct layer/module?
- [ ] No arbitrary new top-level folders?
- [ ] Structure change requires ADR?

---

### 5. Testing Strategy Compliance

- [ ] New code has tests per contract strategy (TDD, BDD, etc)?
- [ ] Minimum coverage threshold met?
- [ ] Test type matches layer (unit for domain, integration for adapters)?

**Violation signal:** PR with zero tests, tests skipped "just this time."

---

### 6. ADR Requirement Check

Flag ADR required when any of the following is detected:

- Switching or adding a framework
- New database or ORM
- New AI provider
- New queue/event system
- Architecture style change
- Security baseline change
- Folder structure overhaul

**Gate:** ADR must exist and be approved before implementation proceeds.

---

### 7. Security Baseline

- [ ] No hardcoded secrets?
- [ ] No credentials in source?
- [ ] Follows contract security baseline (OWASP, etc)?

---

## Verdict Format

```
ARCHITECTURE REVIEW VERDICT
───────────────────────────
Status: PASS | FAIL | PASS WITH FLAGS

Violations:
  - [BLOCKING] <description>
  - [WARNING]  <description>

ADR Required: YES | NO
  → Reason: <if yes>

New Dependencies Flagged:
  - <package> → reason it needs review

Recommendation:
  <summary of what must change before merge>
```

## Common Anti-Patterns (Auto-Fail)

| Anti-Pattern | Why It Fails |
|---|---|
| `import { UserRepo } from '../infrastructure'` in domain layer | Layer boundary violation |
| Adding `lodash` because "it's useful" | Unauthorized dependency |
| 500-line controller with business logic | Architecture style violation |
| New folder `src/utils/helpers/misc/` | Structure drift without ADR |
| PR with 0 tests on business logic | Test strategy violation |
| `const SECRET = "abc123"` in source | Security baseline violation |
| "Generated by AI, looks good" as review | Compiled ≠ Approved |

## Vibe Coding Smell Detector

These phrases in code review justify an immediate FAIL:

- "I'll add tests later"
- "This is just a quick helper"
- "The AI suggested this library"
- "It works, so it's fine"
- "We can refactor later"
- "Just this once"

**None of these are acceptable against an Engineering Contract.**
